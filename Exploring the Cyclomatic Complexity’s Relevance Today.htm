The Relevance of Cyclomatic Complexity in Secure Software Development

Cyclomatic Complexity (CC), a metric that quantifies the number of linearly independent paths through a program’s code, has long been used in assessing software design quality and testing effort. While some critics argue that newer paradigms in software engineering have reduced its utility, CC remains relevant — particularly in the context of developing secure software systems.

1. Cyclomatic Complexity and Security Risk Management

 

High CC often signals a more intricate and potentially fragile codebase. As code complexity increases, so too does the potential for logic flaws, unhandled exceptions, and unintended behaviours — all of which are potential vectors for security vulnerabilities. According to Hassan et al. (2021), high complexity is often positively correlated with the presence of security bugs. In this sense, CC acts as an early warning sign in the identification and management of security risks, aligning directly with the first learning outcome.

 

Moreover, security-focused standards, such as those promoted by OWASP, emphasize the need for simplicity and readability in code — goals that low CC directly supports (OWASP, 2023).

2. Supporting Secure Testing and Verification

 

Cyclomatic Complexity directly informs the minimum number of test cases required to achieve full branch coverage. This becomes particularly critical when verifying code for security properties such as input validation, error handling, and access control logic. As Lee and Shin (2022) note, modules with high CC tend to have lower test coverage and are more resistant to static analysis tools — making them harder to verify and therefore riskier.

 

From a secure development perspective, CC helps developers focus testing and code review efforts on the most critical sections of code, thus ensuring that verification resources are allocated effectively.

3. Practical Relevance in Team-Based Secure Development

 

In modern, often virtualised, development teams, CC metrics can support collaborative quality assurance. When integrated into CI/CD pipelines using tools like SonarQube or GitLab CI, CC provides developers and security engineers with objective, real-time insights into the maintainability and testability of code (Chen et al., 2021).

 

In distributed teams, this fosters a shared understanding of code quality standards and enables earlier detection of risky components — enhancing team effectiveness and secure collaboration.

4. Limitations and Nuanced Application

 

Despite its strengths, CC should not be over-relied upon. It measures control flow but not other security-critical dimensions like data flow, cryptographic usage, or third-party dependency risks. As pointed out by Saini et al. (2023), relying solely on CC may give a false sense of security if broader security assessment practices — such as threat modeling and secure code reviews — are not in place.

 

In addition, modern software often involves asynchronous and reactive programming paradigms, where control flow is non-linear and harder to quantify using traditional CC metrics (Wang et al., 2021).

Conclusion

 

Cyclomatic Complexity remains a relevant and useful metric in today’s secure software development landscape. It helps identify risk-prone modules, prioritise testing, and maintain code quality in distributed teams. However, it should be employed as part of a broader suite of tools and practices aimed at securing code across its lifecycle. In this light, CC continues to support the identification and mitigation of security risks, the evaluation of development methodologies, and the enhancement of team-based secure development workflows.
